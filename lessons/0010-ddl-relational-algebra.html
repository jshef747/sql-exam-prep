שיעור 10 · חלק א' של המבחן — DDL ואלגברה יחסית

הגדרת טבלאות (DDL) ו‑SQL ↔ אלגברה יחסית

השיעור האחרון, והוא שונה: כאן לא כותבים שאילתות גדולות אלא עונים על שאלות חלק א' של המבחן — שאלות בחירה (MCQ) ומושגיות. שני נושאים: (א) הגדרת טבלה עם מפתח ראשי, מפתח זר, פעולות שמירת קישור (on delete cascade/set null) ואילוצי assertion/check; (ב) לקרוא ביטוי באלגברת היחסים (σ π ⋈ × − ÷ ρ), לזהות את ה‑SQL השקול לו ולהפך, ולתרגם RA→SQL.

🎯 היעד: לקרוא נכון משפט create table ולדעת מה הוא מבטיח; להבין ש‑on delete cascade זורם מהאב לבן בלבד; לזהות באיזו תשובה ביטוי RA שקול לשאילתת SQL נתונה (וההפך) — ולענות נכון על שאלות חלק א' Q6 ו‑Q4 של המבחן.
↩️ עד כה (שיעורים 1–9): שליפה, צירופים, like, הקבצה, MAX/MIN, צירוף חיצוני, תת‑שאילתות, חלוקה ("לכל"), וספירה/השוואה. אלו שאלות חלק ב' — כתיבת שאילתות. עכשיו סוגרים את הקורס עם חלק א': שאלות תיאוריה ו‑MCQ על DDL ואלגברה יחסית. כל מה שלמדנו על σ/π/⋈/− חוזר כאן, הפעם בכיוון ההפוך — מ‑SQL לאלגברה.

סכמה מלאה ב‑מילון המונחים והסכמות ←. בשאלות חלק א' של המבחן מופיעות גם סכמות מופשטות — יחסים בשמות כמו r(A,B,C) ו‑s(C,D,E), או task/mech — שעליהן אי אפשר "להריץ" כי אין להן נתונים; אותן פותרים בראש (MCQ).

1DDL — הגדרת טבלה: create table

עד כה רק שלפנו מטבלאות קיימות. DDL (Data Definition Language) זה איך מגדירים את הטבלה מלכתחילה — שם, עמודות, טיפוסים, ואילוצי שלמות. הצורה (מצגת תיכון בסיסי נתונים, פרק 4 — שלמות קישור):

create table customer
  (customer_name  char(20) not null,
   street         char(30),
   customer_city  char(30),
   primary key    (customer_name))

כל שורה היא עמודה: שם + טיפוס (למשל char(20), integer, date). not null אוסר ערך ריק בעמודה. בסוף נכתבים אילוצי המפתח.

טיפוסמשמעות
char(n)מחרוזת באורך עד n תווים.
integer · smallintמספר שלם.
numeric(p,s) · realמספר עם נקודה עשרונית.
dateתאריך.
not nullמניעת ערך דמה (null) בעמודה זו.

2מפתח ראשי ומפתח זר

מפתח ראשי (primary key) — עמודה (או צירוף עמודות) שמזהה כל שורה ביחידות. כל עמודה ששייכת למפתח קביל היא אוטומטית not null (מצגת פרק 4).

מפתח זר (foreign key) — עמודה בטבלה שלנו שמצביעה על מפתח (קביל) בטבלה אחרת. הוא מבטיח שלמות קישור (referential integrity): לא תהיה שורה "מדולדלת" (dangling) שמצביעה על מישהו שלא קיים. דוגמה מלאה:

create table deposit
  (branch_name     char(15),
   account_number  char(10) not null,
   customer_name   char(20) not null,
   balance         integer,
   primary key (account_number, customer_name),
   foreign key (branch_name)   references branch,
   foreign key (customer_name) references customer)

שתי השורות האחרונות אומרות: לכל ערך ב‑branch_name חייב להתקיים סניף תואם בטבלת branch, ולכל customer_name חייב להתקיים לקוח ב‑customer. אם תנסה להכניס שורה עם branch_name שלא קיים ב‑branch — ההכנסה תידחה.

💡 בכתיב המלא כותבים גם את עמודת היעד: foreign key (proj_no) references mech (proj_no) — "העמודה proj_no שלי מצביעה על העמודה proj_no בטבלה mech". כשהשם זהה אפשר לקצר, אבל המבחן לרוב מציג את הכתיב המלא.

3⭐ פעולות שמירת קישור: on delete cascade / set null

מה קורה כשמוחקים שורה מטבלת האב (הטבלה שאליה המפתח הזר מצביע), בזמן שיש שורות בן שמצביעות עליה? בלי הוראה — המחיקה תידחה (כדי לא ליצור שורות מדולדלות). אפשר לבקש התנהגות אחרת:

הוראהמה קורה בעת מחיקת שורת האב
on delete cascadeכל שורות הבן שמצביעות על האב נמחקות גם הן (מחיקה "מתגלגלת").
on delete set nullבשורות הבן, עמודת המפתח הזר מתעדכנת ל‑null (השורות נשארות, רק מאבדות את הקישור).

נניח שהגדרנו בטבלת task (הבן):

foreign key (proj_no) references mech (proj_no) on delete cascade

כאן mech הוא האב (הטבלה המוצבעת), ו‑task הוא הבן (הטבלה עם המפתח הזר). on delete cascade אומר: כשמוחקים שורה מ‑mech, כל שורות task שהצביעו עליה נמחקות.

⚠️ הכיווניות היא מלכודת הבחינה (Q6): cascade זורם מהאב (mech) אל הבן (task) בלבד — לעולם לא הפוך. מחיקת שורה מ‑task לא מוחקת דבר מ‑mech (אין שום סיבה: mech לא מצביע על task). זכרו: המחיקה מתגלגלת מהמוצבע אל המצביע.

הערה: create table עם שם טבלה חדש רץ במנוע (▶) ומחזיר "✓ בוצע". on delete cascade, create assertion ו‑check אינם נתמכים במנוע — לכן בשיעור הם מסומנים בלי כפתור הרצה.

4אילוצי check ו‑create assertion

מעבר למפתחות, אפשר לאכוף חוקים כלליים על הנתונים. create assertion N check (Ψ) יוצר הצהרה בשם N שמבטיחה את התנאי Ψ בכל רגע (מצגת בקרת שלמות הנתונים — אכיפת שלמות נתונים ב‑SQL). הטריק הקבוע: כותבים את התנאי כ‑"לא קיימת שורה שמפרה אותו" — not exists (...).

דוגמה 1 — "נכסי כל הסניפים ידועים" (אין null)

create assertion assets1
check ( not exists
        ( select * from branch
          where assets is null ))

קוראים: "ההצהרה תקפה כל עוד לא קיים סניף שה‑assets שלו null". זה בדיוק כמו not null על העמודה — אבל מנוסח כאילוץ גלובלי.

דוגמה 2 — אכיפת תלות פונקציונלית city → branch_name

create assertion assets2
check ( not exists
        ( select city from branch
          group by city
          having count(distinct branch_name) > 1 ))

"לא קיימת עיר שיש בה יותר מסניף אחד" — בדיוק האכיפה של "עיר קובעת סניף יחיד". שימו לב לתבנית: ההפרה (עיר עם >1 סניף) נתפסת ב‑group by ... having count(...) > 1, וה‑not exists דורש שהיא לא תתרחש.

דוגמה 3 — "סך ההלוואות בסניף אינו עולה על סך היתרות"

create assertion sum_constraint
check ( not exists
        ( select * from branch
          where ( select sum(amount)  from borrow
                  where borrow.branch_name  = branch.branch_name )
              > ( select sum(balance) from deposit
                  where deposit.branch_name = branch.branch_name ) ))

"לא קיים סניף שבו סך ההלוואות גדול מסך היתרות". כל אילוץ assertion בנוי כך: נסחו את ההפרה כשאילתה, ועטפו ב‑not exists.

💡 check אפשר לכתוב גם בתוך הגדרת טבלה (אילוץ על שורה בודדת, למשל check (age >= 18)), אבל create assertion הוא אילוץ גלובלי שיכול לערב כמה טבלאות (כמו דוגמה 3). drop assertion N מוחק את האכיפה.

5אלגברה יחסית — מילון האופרטורים

אלגברת היחסים היא ה"מתמטיקה" שמאחורי SQL. כל ביטוי בה ניתן לתרגום ל‑SQL ולהפך. אלו האופרטורים שתפגשו בחלק א' (מצגת תיאור שפת SQL, פרק 3):

סימןשםמה עושהב‑SQL
σבחירה (selection)סינון שורות לפי תנאיwhere
πהטלה (projection)בחירת עמודות (ומסיר כפילויות)select distinct
×מכפלה קרטזיתכל שילוב שורות משתי טבלאותfrom a, b
⋈צירוף טבעי (join)חיבור לפי עמודות בעלות שם משותףnatural join / join ... on
−הפרש (difference)שורות שב‑a ואינן ב‑bnot in / not exists / except
∪איחוד (union)שורות שב‑a או ב‑bunion
÷חילוק (division)"לכל" — מי שמתאים לכל ערכי bnot exists ... not exists (שיעור 8)
ρשינוי שם (rename)כינוי לטבלה/עמודה — מאפשר צירוף‑עצמיas (alias)

הקשר הבסיסי שכבר ראינו במילון: שאילתת SQL בסיסית select A from r where c שקולה בדיוק ל‑πA(σc(r)).

σ → where· π → select distinct· ⋈ → natural join· − → not in / except

6תרגום RA → SQL (ולהפך)

הכלל: קוראים את הביטוי מבפנים החוצה. כל σ הופך ל‑where, כל π ל‑select, כל ⋈ ל‑join, וכל − ל‑not in / not exists. דוגמה פשוטה (סכמת ביטוח הרכב):

πfname,lname ( σcity='Haifa' ( person ) )

קוראים: סנן (σ) את person לעיר Haifa, ואז הטל (π) את העמודות fname, lname:

select fname, lname
from   person
where  city = 'Haifa'

עם צירוף וחיסור

הביטוי שראינו בשיעור 7 (השאלה ממבחן 2023): πdamage(Accident) − πdamage(σa2.damage>a1.damage(Accident × ρa2(Accident))) — "כל הנזקים פחות כל נזק שיש מעליו נזק גדול יותר" = הנזק הגבוה ביותר. החיסור (−) הופך ל‑not exists/not in, וה‑ρ הוא הכינוי לצירוף‑עצמי. (השלמנו שם את התרגום; כאן רק נזכיר את הכיוון.)

💡 − (הפרש) הוא הלב של שאלות "שלא"/"שאין": ΠA(r) − ΠA(σc(...)) = "כל ה‑A, פחות ה‑A שמקיימים את c" = "ה‑A שלא מקיימים c" ⟵ where A not in (select A ... where c). את התבנית הזו פוגשים ב‑Q4 של המבחן.

תרגול — כתבו ובדקו את עצמכם ✍️

נסו בעצמכם לפני "הצג פתרון". הרמה עולה: חימום → תרגול → שאלות ממבחנים אמיתיים. המטרה שתדעו לפתור כל שאלה שבמבחנים. השאלות החיות (RA→SQL) נבדקות אוטומטית על סכמת הביטוח; שאלות ה‑MCQ והסכמות המופשטות (r,s / task,mech) — נפתרות בראש.

חימום ותרגול

תרגיל 1 (RA → SQL, חי) — תרגמו לשאילתת SQL את הביטוי הבא והריצו אותה: 
π fname,lname ( σ city='Haifa' ( person ) )
select fname, lname
from   person
where  city = 'Haifa'

σ = ה‑where, π = ה‑select. מבפנים החוצה: קודם הסינון, ואז בחירת העמודות. (בנתוני הדוגמה מוחזרות 6 שורות.)

תרגיל 2 (RA → SQL, חי — צירוף) — תרגמו לשאילתת SQL את הביטוי, והריצו. (person ⋈ owns ⋈ car = צירוף טבעי לאורך id ואז license.) 
π fname,lname ( σ brand='Tesla' ( person ⋈ owns ⋈ car ) )
select fname, lname
from   person join owns on person.id = owns.id
             join car  on owns.license = car.license
where  car.brand = 'Tesla'

— אותו דבר עם צירוף טבעי / פסיק ו‑where:

select fname, lname
from   person, owns, car
where  person.id = owns.id
  and  owns.license = car.license
  and  car.brand = 'Tesla'

כל ⋈ הוא join ... on לאורך העמודה המשותפת, וה‑σ הוא ה‑where. (בנתוני הדוגמה מוחזרות 4 שורות.)

תרגיל 3 (DDL, חי) — כתבו create table ליחס חדש בשם project עם העמודות proj_no (integer, מפתח ראשי), pname (char(40), חובה), ו‑budget (integer). הריצו — אמורים לקבל "✓ בוצע".
create table project
  (proj_no  integer,
   pname    char(40) not null,
   budget   integer,
   primary key (proj_no))

טיפוס לכל עמודה, not null ל‑pname, והמפתח הראשי בסוף. שם הטבלה חדש (לא מהסכמה) — לכן רץ במנוע ומחזיר "בוצע".

תרגיל 4 (מפתח זר — קריאה) — נתון: foreign key (garage_id) references garage (garage_id) on delete set null בטבלת accident. מה קורה כשמוחקים מוסך מטבלת garage שטיפל ב‑3 תאונות?

(שאלת קריאה — אין צורך להריץ.)

שלוש שורות התאונה לא נמחקות — אבל עמודת garage_id שלהן הופכת ל‑null ("התאונה קיימת, אך כרגע לא משויכת לאף מוסך"). אם זה היה on delete cascade — שלוש התאונות היו נמחקות. אם לא הייתה הוראה כלל — מחיקת המוסך הייתה נדחית כל עוד יש תאונות שמצביעות עליו.

שאלות ממבחנים אמיתיים 🎓

עכשיו ברמת הבחינה — אלו שאלות חלק א' MCQ אמיתיות. היעזרו ב‑מילון הסכמות אם צריך.

🎓 תרגיל 5 (Q6) — בהגדרת SQL של יחס בשם task מופיעה השורה הבאה: 
foreign key (proj_no) references mech (proj_no) on delete cascade
איזה מהמשפטים הבאים אינו נכון?

📄 מקור: מבחן 131112 · 2023 · סמסטר 2 · מועד 1 (קובץ …_16208) · חלק א' שאלה 6

התשובה הלא נכונה היא ב'. on delete cascade זורם מהאב (mech, המוצבע) אל הבן (task, המצביע) — מחיקת שורה מ‑mech מוחקת את שורות task התלויות (כך א' נכון). אבל הכיוון ההפוך לא קיים: מחיקת שורה מ‑task לא משפיעה על mech (אין סיבה — mech לא מצביע על task). לכן ב' שגוי.

הסבר על השאר: ג' נכון — העמודה proj_no ב‑mech חייבת להיות מפתח קביל, כי מפתח זר יכול להצביע רק על מפתח קביל. ד' נכון — זו בדיוק שלמות הקישור: לא ניתן להכניס ל‑task ערך proj_no שלא קיים ב‑mech, אחרת תיווצר שורה מדולדלת.

🎓 תרגיל 6 (Q4) — נתונים שני יחסים r ו‑s בעלי תבניות R=(A,B,C) ו‑S=(C,D,E) בהתאמה (תחום כל התכונות integer). נתונה השאילתה: 
select distinct A
from   r
where  A not in (select A
                 from   r natural inner join s
                 where  E > 3)
איזו מהתשובות הבאות (באלגברת היחסים) שקולה לה?

📄 מקור: מבחן 131112 · 2023 · סמסטר 2 · מועד 1 (קובץ …_16208) · חלק א' שאלה 4. (סכמה מופשטת → MCQ, לא ניתן להריץ.)

התשובה הנכונה היא א'. פירוק השאילתה:

  • תת‑השאילתה הפנימית: select A from r natural inner join s where E>3 = ΠA( σE>3(r ⋈ s) ) — ערכי ה‑A שמשתתפים בצירוף עם שורת s שבה E>3.
  • החיצונית: distinct A מ‑r ש‑A שלהם אינו בקבוצה הזו ⟵ חיסור: ΠA(r) − ΠA( σE>3(r ⋈ s) ).

תשובה א' מנסחת בדיוק את החלק השני: ΠA,C(r) ⋈ σE>3(s) — הצירוף הטבעי הוא לאורך C (התכונה המשותפת היחידה), והטלת r מראש ל‑A,C לא משנה אילו ערכי A שורדים (העמודה B כלל לא משתתפת בצירוף או בסינון). לכן ΠA(ΠA,C(r) ⋈ σE>3(s)) = ΠA(σE>3(r ⋈ s)), וכל הביטוי שקול לשאילתה.

למה השאר שגויות:

  • ב' נותנת את הקבוצה החיובית (ה‑A שכן בצירוף) — בדיוק ההיפך מ‑not in.
  • ד' כותבת ΠA(σE>3(s)) — אבל ל‑s אין עמודה A כלל, אז ההטלה הזו אינה חוקית/אינה מה שהשאילתה עושה.
  • ג' היא חילוק (÷ = "לכל"), ואילו השאילתה היא חיסור (not in) — סוג שונה לגמרי.

הקשר לשיעור: not in (subquery) ⟷ − (הפרש), natural inner join ⟷ ⋈, where E>3 ⟷ σE>3. זיהוי שלושת אלו מוביל ישר לתשובה א'.

בדיקת הבנה (משוב מיידי) 🧠

1. בטבלה task מוגדר foreign key (proj_no) references mech (proj_no) בלי שום הוראת on delete. מנסים למחוק שורה מ‑mech שיש לה שורות task שמצביעות עליה. מה קורה?
